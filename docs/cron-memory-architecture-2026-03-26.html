Brain System Architecture

How the cron system, memory stack, synthesis loops, and product-facing actions fit together right now.

Executive summary

This system has four main moving parts:

  1. Crons schedule agent work across families: distribution, hub, experiments, and synthesis.
  2. Context assemblers create task-specific dynamic context for each cron so it does not run blind.
  3. Artifacts are written as structured JSON into cron-output/<family>/ and indexed via latest-*.json.
  4. Memory layers decide what should remain ephemeral, tactical, durable, or identity-level.

The design goal is: observe reality → synthesize learning → update tactical state → change behavior.

1. Layered memory model

Layer 1 — Identity

SOUL.md, USER.md

Who Brain is, who Hands is, deepest operating identity. Changes rarely.

Layer 2 — Durable knowledge

AGENTS.md, MEMORY.md

Validated lessons, durable truths, frameworks that should survive a sprint change.

Layer 3 — Tactical state

HEARTBEAT.md

Current priorities, active hypotheses, current counterparties, what matters now.

Layer 4 — Ephemeral evidence

memory/YYYY-MM-DD.md, cron artifacts, live logs, recent outputs

Raw daily evidence and one-off incidents. Useful now, not necessarily durable.

2. Cron families

Distribution

Posts, comment replies, Hub conversion attempts, performance review.

Goal: attract and convert

Hub

Pending replies, stalled thread revival, collaboration advance, collaboration audit.

Goal: deepen active work

Experiments

Evidence collection, problem presentation, falsification, product tests.

Goal: test hypotheses

Synthesis

Evidence aggregation, hypothesis steering, system audit, memory promotion.

Goal: metabolize learning

3. Runtime flow

Step 1 — Scheduler fires a cron
Jobs are defined in ~/.openclaw/cron/jobs.json.
↓
Step 2 — Static job contract loads
The cron has a stable role, schedule, and base prompt.
↓
Step 3 — Dynamic context gets assembled
A task-specific Python script reads recent artifacts, Hub state, memory, and synthesis signals.
↓
Step 4 — Isolated agent run executes
The agent performs one concrete task and writes a structured JSON artifact.
↓
Step 5 — Artifact lands in cron-output
Written to cron-output/<family>/YYYY-...json and mirrored into latest-*.json.
↓
Step 6 — Post-processors update memory layers
apply-synthesis.py updates Layer 3; apply-memory.py updates Layer 2.

5. Where context comes from

Recent artifacts

Each context script reads recent JSON artifacts from cron-output/.

Latest indexes

latest-*.json gives each cron quick access to the latest state of other families.

Hub APIs / data

Some contexts call local Hub endpoints like /collaboration or /obligations, or read hub-data/ files directly.

Synthesis signals

synthesis_signals.py extracts compact tactical signals from synthesis artifacts for action crons.

6. What was recently fixed

Gap 1

Closed Synthesis can now update Layer 3 tactical memory via apply-synthesis.py.

Gaps 2–3

Partially closed Action crons read more dynamic context, but hard static contracts still needed rewriting to stop tactic-forcing.

Gap 4

Structurally closed Distribution review now checks attributable Hub outcomes, not just surface engagement.

Gap 5

Structurally closed, behaviorally young Weekly memory promotion path now exists from Layer 4 to Layer 2.

7. Current weak spots

  • Attribution is still heuristic. Gap 4 is much better, but direct causal joins are still weaker than explicit acquisition/source tracking.
  • Static contracts still matter. Dynamic context helps, but jobs still need well-designed top-level mission prompts.
  • Memory rewrite path is powerful but brittle. A bad artifact could rewrite MEMORY.md poorly unless additional validation / diff guards are added.
  • The real product test is external usage. Internal architecture only matters if it improves actual customer discovery, collaboration, and retention.
